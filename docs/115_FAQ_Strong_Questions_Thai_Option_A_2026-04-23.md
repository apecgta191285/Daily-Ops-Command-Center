# FAQ — Strong Committee Questions (Thai) for Option A
วันที่: 23 เมษายน 2026

## 1) ระบบนี้คืออะไรแน่
ระบบนี้เป็น internal web app สำหรับการจัดการงานประจำวันของห้องปฏิบัติการคอมพิวเตอร์หลายห้องในมหาวิทยาลัยเดียว โดยเชื่อม checklist, incident, dashboard, history, และ governance เข้าด้วยกัน

## 2) ทำไมต้องมี 3 roles เท่านั้น
เพราะกรณีศึกษาจริงมีทีมเล็ก และหน้าที่หลักแบ่งได้ชัดเป็นผู้ดูแลระบบ, ผู้ติดตามงาน, และผู้ปฏิบัติงานหน้างาน การมี 3 roles ทำให้ระบบไม่ซับซ้อนเกินความจำเป็นและยังครอบคลุม workflow หลักได้

## 3) ทำไมต้องเพิ่ม room เข้ามา
เพราะถ้าดูงานตามช่วงเวลาอย่างเดียว ระบบจะตอบได้แค่ว่ามีงานหรือมีปัญหา แต่ยังไม่ตอบชัดว่าปัญหาเกิดที่ห้องไหน การเพิ่ม room ทำให้ระบบสอดคล้องกับกรณีศึกษาหลายห้องมากขึ้น

## 4) ถ้าเครื่องเสียทำยังไง
ในเวอร์ชันนี้ นักศึกษาสามารถสร้าง incident แล้วระบุ room context พร้อม optional equipment reference เช่น `PC-12` ได้ เพื่อให้ผู้ดูแลห้องรู้ว่าอุปกรณ์ไหนเกี่ยวข้อง แต่ระบบยังไม่ถึงขั้น machine registry

## 5) แล้วทำไมไม่ทำ machine registry ไปเลย
เพราะ machine registry ต้องเปิด domain ใหม่ ทั้ง schema, relationship, และ workflow การพยายามยัดเข้ามาใน wave เดียวจะทำให้ขอบเขตเละและอธิบายยาก จึงเลือกทำ room-centered operations ให้ถูกต้องก่อน

## 6) ระบบนี้ต่างจาก checklist app ทั่วไปยังไง
ความต่างคือระบบนี้ไม่ได้จบแค่ checklist แต่เชื่อมไปถึง incident handoff, dashboard/workboard, history, และ governance ทำให้การทำงานของทีมอยู่ใน workflow เดียว

## 7) ระบบนี้เป็น production-grade หรือยัง
ยังไม่ควรอ้างแบบนั้น สถานะที่ตรงกว่า คือ strong capstone / disciplined MVP+ / credible internal prototype

## 8) ทำไมไม่ทำ multi-tenant หรือหลายองค์กร
เพราะกรณีศึกษานี้อยู่ในมหาวิทยาลัยเดียวและทีมเดียว การเพิ่มหลายองค์กรตอนนี้ไม่ช่วยให้ product truth ชัดขึ้น และจะทำให้ระบบซับซ้อนเกินโจทย์

## 9) optional equipment reference มีประโยชน์อะไรถ้าไม่ได้เป็น machine registry
มันช่วยให้ incident มีบริบทมากขึ้น เช่น บอกได้ว่าปัญหาเกี่ยวกับ `PC-12` หรือ `Projector Front` โดยไม่ต้องสร้างระบบ asset management เต็มรูปแบบทันที

## 10) ถ้าจะต่อยอดหลังจากนี้ ควรไปทางไหน
ถ้าจะต่อยอดจริง ควรเปิด wave ใหม่อย่างมีวินัย เช่น machine registry foundation หรือ room operations maturity ที่ลึกขึ้น แต่ไม่ควรแกล้งทำเหมือนของเหล่านั้นอยู่ในเวอร์ชันปัจจุบันแล้ว

## 11) ทำไมยังต้องมี admin ถ้า supervisor ก็ดู dashboard ได้
เพราะ admin มีหน้าที่ดูแล governance เช่น template และบัญชีผู้ใช้ ซึ่งเป็นคนละความรับผิดชอบกับ supervisor ที่เน้นติดตามงานปฏิบัติการรายวัน

## 12) ทำไมต้องมี history และ print surfaces
เพราะในงานจริงและในการเดโม เราต้องย้อนดูได้ว่าห้องไหนถูกตรวจอะไรไปแล้ว และ incident ไหนเคยเกิดขึ้นหรือปิดไปแล้ว การมี print-friendly surfaces ยังช่วยใช้เป็น evidence สำหรับ review ได้ด้วย

## 13) ถ้ากรรมการถามว่า “ระบบนี้ยังไม่สมบูรณ์ใช่ไหม”
ตอบได้ว่าใช่ ระบบนี้ยังไม่สมบูรณ์ทุกมิติ แต่ขอบเขตที่ตั้งใจทำในรอบนี้ทำได้ถูกต้องและอธิบายได้จริง คือ room-centered daily operations สำหรับหลายห้อง โดยไม่หลอกตัวเองว่าเป็น platform ใหญ่เกินความจริง
