# **1\. Product Vision**

สร้างเว็บแอปสำหรับทีมงานขนาดเล็กที่ต้องมีงานเช็กลิสต์ประจำวันและการติดตามเหตุการณ์ผิดปกติ โดยลดการพึ่งพากระดาษหรือการสื่อสารในแชตที่ตรวจสอบย้อนหลังยาก และเปลี่ยนให้เป็น flow ที่ติดตามได้จากหน้าเว็บเดียว

# **2\. Target Users**

| Persona | ลักษณะผู้ใช้ | สิ่งที่ผู้ใช้ต้องการ |
| :---: | ----- | ----- |
| Admin | ผู้ตั้งค่าระบบและโครง checklist | สร้างแม่แบบและดูภาพรวมระบบ |
| Supervisor | หัวหน้าทีมหรือผู้ตรวจงาน | เห็นงานครบ/ไม่ครบและปัญหาคงค้าง |
| Staff | ผู้ปฏิบัติงานประจำวัน | เช็กงานที่ต้องทำและแจ้งเหตุผิดปกติได้ง่าย |

# **3\. Core Jobs-to-be-Done**

* ก่อนเริ่ม/ระหว่าง/หลังจบงาน ผู้ปฏิบัติงานต้องรู้ว่ามีรายการใดต้องตรวจหรือทำให้ครบ  
* เมื่อพบความผิดปกติ ผู้ใช้ต้องแจ้งเหตุและส่งต่อข้อมูลให้หัวหน้าทีมติดตามได้ทันที  
* หัวหน้าทีมต้องเห็นภาพรวมของวันนั้นได้อย่างรวดเร็วโดยไม่ไล่ดูหลายแหล่งข้อมูล

# **4\. Why This App Exists**

* เพราะงานประจำวันจำนวนมากไม่ซับซ้อนพอจะต้องใช้ระบบ enterprise แต่ก็ไม่ควรปล่อยให้ทำแบบไม่มีหลักฐาน  
* เพราะทีมเล็กมักไม่มีเครื่องมือที่ตรง use case และตั้งค่าง่ายในบริบทภาษาไทย  
* เพราะโจทย์เชิงวิชาการของโครงงานนี้ต้องการระบบงานจริงที่วัดผลและเดโมได้ ไม่ใช่ dashboard ลอย ๆ หรือแอปเล่น ๆ

# **5\. Why Not Just Use Global Tools?**

* **Brutal Truth:** ถ้าเป็นองค์กรจริงที่มีงบและต้องการระบบ mature มาก ผู้ใช้จำนวนมากอาจเลือกเครื่องมือระดับโลกอยู่แล้ว  
* **Reason to build this project:** ระบบนี้โฟกัสที่ทีมเล็ก, setup ง่าย, ภาษาไทย, และ use case แคบที่เข้าใจได้ในเวลาอันสั้น  
* **Academic value:** การสร้างระบบเองทำให้ควบคุม baseline, evidence, และการประเมินผลเชิงระบบได้ชัดกว่าการหยิบ SaaS มาใช้เฉย ๆ

# **6\. Product Positioning**

| มิติ | ระบบของเรา | เครื่องมือ global |
| :---: | ----- | ----- |
| Scope | แคบและตรง use case daily ops สำหรับทีมเล็ก | กว้างและรองรับ use case ระดับองค์กร |
| Setup | ตั้งค่าไม่เยอะใน v1 | มักมีตัวเลือกจำนวนมากและ setup หนักกว่า |
| Language/Context | ออกแบบให้ใช้คำและ flow ภาษาไทยได้ง่าย | หลายตัวเริ่มจากบริบททั่วไปหรือภาษาอังกฤษ |
| Project Goal | ใช้เป็นระบบต้นแบบเพื่อพิสูจน์แนวทางและประเมินผล | ใช้เพื่อผลิตงานจริงใน production |

# **7\. Success Criteria**

* ผู้ใช้ทำ checklist รายวันสำเร็จได้ใน flow เดียวโดยไม่สับสน  
* incident ถูกบันทึกและติดตามสถานะได้ครบตามข้อมูลขั้นต่ำ  
* supervisor มองภาพรวมของวันนั้นได้จาก dashboard เดียว  
* baseline แบบ manual/chat มีข้อจำกัดชัดเมื่อเทียบกับระบบ A-lite  
* เอกสารและ implementation พูดตรงกันว่าเป้าหมายคือ demo-ready MVP ไม่ใช่ enterprise replacement
