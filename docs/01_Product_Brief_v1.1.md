# **1\. Product Vision**

สร้างเว็บแอปสำหรับทีมดูแลห้องคอมของมหาวิทยาลัยที่ต้องมีงานตรวจเช็กประจำวัน การแจ้งเหตุผิดปกติ และการติดตามงานของหัวหน้าทีม โดยลดการพึ่งพากระดาษหรือการสื่อสารในแชตที่ตรวจสอบย้อนหลังยาก และเปลี่ยนให้เป็น flow ที่ติดตามได้จากหน้าเว็บเดียว

# **2\. Target Users**

| Persona | ลักษณะผู้ใช้ | สิ่งที่ผู้ใช้ต้องการ |
| :---: | ----- | ----- |
| Admin | ผู้ดูแล template และ account lifecycle ของทีม | สร้างแม่แบบ, ดูแล roster, และคุม baseline ของระบบ |
| Supervisor | หัวหน้าทีมห้องคอม | เห็น lane ของวัน, ปัญหาคงค้าง, และ follow-up pressure |
| Staff | เจ้าหน้าที่ประจำวันหรือ duty staff | เปิดห้อง, ตรวจระหว่างวัน, ปิดห้อง, และแจ้งเหตุผิดปกติได้ง่าย |

# **3\. Core Jobs-to-be-Done**

* ก่อนเปิดห้อง ระหว่างวัน และก่อนปิดห้อง ผู้ปฏิบัติงานต้องรู้ว่ามีรายการใดต้องตรวจหรือทำให้ครบ  
* เมื่อพบความผิดปกติ เช่น workstation ไม่พร้อม, network ใช้งานไม่ได้, printer มีปัญหา, หรือสภาพห้องไม่พร้อม ผู้ใช้ต้องแจ้งเหตุและส่งต่อข้อมูลให้หัวหน้าทีมติดตามได้ทันที  
* หัวหน้าทีมต้องเห็นภาพรวมของวันนั้นได้อย่างรวดเร็วโดยไม่ไล่ดูหลายแหล่งข้อมูล

# **4\. Why This App Exists**

* เพราะงานประจำวันจำนวนมากไม่ซับซ้อนพอจะต้องใช้ระบบ enterprise แต่ก็ไม่ควรปล่อยให้ทำแบบไม่มีหลักฐาน  
* เพราะทีมเล็กมักไม่มีเครื่องมือที่ตรง use case ของห้องคอมและตั้งค่าง่ายในบริบทภาษาไทย  
* เพราะโจทย์เชิงวิชาการของโครงงานนี้ต้องการระบบงานจริงที่วัดผลและเดโมได้ ไม่ใช่ dashboard ลอย ๆ หรือแอปเล่น ๆ

# **5\. Why Not Just Use Global Tools?**

* **Brutal Truth:** ถ้าเป็นองค์กรจริงที่มีงบและต้องการระบบ mature มาก ผู้ใช้จำนวนมากอาจเลือกเครื่องมือระดับโลกอยู่แล้ว  
* **Reason to build this project:** ระบบนี้โฟกัสที่ทีมเล็ก, setup ง่าย, ภาษาไทย, และ use case แคบที่เข้าใจได้ในเวลาอันสั้น  
* **Academic value:** การสร้างระบบเองทำให้ควบคุม baseline, evidence, และการประเมินผลเชิงระบบได้ชัดกว่าการหยิบ SaaS มาใช้เฉย ๆ

# **6\. Product Positioning**

| มิติ | ระบบของเรา | เครื่องมือ global |
| :---: | ----- | ----- |
| Scope | แคบและตรง use case ของทีมดูแลห้องคอมมหาวิทยาลัย | กว้างและรองรับ use case ระดับองค์กร |
| Setup | ตั้งค่าไม่เยอะใน v1 | มักมีตัวเลือกจำนวนมากและ setup หนักกว่า |
| Language/Context | ออกแบบให้ใช้คำและ flow ของงานห้องคอมได้ง่าย | หลายตัวเริ่มจากบริบททั่วไปหรือภาษาอังกฤษ |
| Project Goal | ใช้เป็นระบบต้นแบบเพื่อพิสูจน์แนวทางและประเมินผล | ใช้เพื่อผลิตงานจริงใน production |

# **7\. Success Criteria**

* ผู้ใช้ทำ lab opening / during-day / room closing checklist ได้ใน flow เดียวโดยไม่สับสน  
* incident ถูกบันทึกและติดตามสถานะได้ครบตามข้อมูลขั้นต่ำ  
* supervisor มองภาพรวมของวันนั้นได้จาก dashboard เดียวและรู้ว่า lane ไหนยังค้าง  
* baseline แบบ manual/chat มีข้อจำกัดชัดเมื่อเทียบกับระบบ A-lite  
* เอกสารและ implementation พูดตรงกันว่าเป้าหมายคือ demo-ready MVP ไม่ใช่ enterprise replacement
